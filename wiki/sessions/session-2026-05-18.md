## Completed

### Pi SDK Sandbox postMessage Debugging

1. **Root cause**: `sendSDKMessage` in Pi SDK calls `window.parent.postMessage(l, f)` with hardcoded target origin `"https://sandbox.minepi.com"`. When running top-level (no iframe parent), `window.parent === window` and origin is `https://axiomid.app` → origin mismatch error.

2. **Strategy 1 — `window.postMessage` override** (primary, untested):
   - `Window.prototype.postMessage` has `writable: true` but `configurable: false`
   - Assignment `window.postMessage = fn` creates an own property on `window` that shadows the prototype
   - Chrome does NOT block assignment via WindowProxy (unlike `Object.defineProperty`)
   - Custom handler intercepts, re-writes origin to `"https://sandbox.minepi.com"` if message matches SDK pattern

3. **Strategy 2 — Instance-level `Object.defineProperty(window, "parent")`** (tested, failed):
   - Chrome WindowProxy likely blocks `defineProperty` on `window` for `parent`
   - Error persisted in production deploy

4. **Strategy 3 — Prototype-level `Window.prototype.parent` getter** (tested, failed):
   - Extension injects `window.parent` as a non-configurable own property on `window`
   - Prototype override is shadowed; never reached

### CSP & SDK Loading

- Added `'unsafe-inline'` to `script-src` (required for Pi extension injected `<script>` tags)
- Added preconnect `Link` header for `https://sandbox.minepi.com`
- Implemented hybrid `ensurePiSdk()`: checks extension-injected `window.Pi` first, falls back to CDN with 3 retries
- Never deletes existing `window.Pi` (avoids stream desync race with extension)
